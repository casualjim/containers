# Design: Chisel Ubuntu 26.04 Migration

## Summary

Single-version bump of all chisel-based container images from Ubuntu 25.10 (Questing) to Ubuntu 26.04 LTS (Resolute Raccoon). Updates `UBUNTU_RELEASE` and `CHISEL_VERSION` variables, syncs upstream chisel slices from the `ubuntu-26.04` branch, rewrites custom LLVM libc++ slices from LLVM 20 to LLVM 22, and updates all documentation references. No target changes, no CI changes, no dual builds — 25.10 support is dropped entirely.

## Requirements

- All 10 chisel-based images must build and run on Ubuntu 26.04
- Library resolution: libstdc++6, libssl3t64, libc++1, libc++abi1, llvm-libunwind1, zlib1g, libgcc-s1
- Tesseract OCR + language packs must install and function
- Ladybug shared library must load correctly
- Bun runtime must pass `bun --version`
- Cross-architecture builds (amd64 + arm64) must succeed
- No dual-version builds — drop 25.10 support entirely

## Decisions

### libc++abi1 → llvm-libunwind1 dependency: Follow deb metadata

**Ambiguity**: The LLVM 20 pattern has `libc++abi1-20` depending on `libunwind-20` in its essentials (`chisel-releases/slices/libc++abi1-20.yaml:10`). For 26.04, the package `libc++abi1` (from `llvm-toolchain-22`) shows deb-level dependencies as only `libc6` + `libgcc-s1` — no explicit `llvm-libunwind1` dependency.

**Explored**:
- Option A: Follow deb metadata strictly — `libc++abi1_libs` essentials = `[libc6_libs, libgcc-s1_libs]`. `llvm-libunwind1_libs` is a peer under `libc++1_libs`. Matches Ubuntu package manager.
- Option B: Follow LLVM 20 pattern — `libc++abi1_libs` essentials include `llvm-libunwind1_libs`. Preserves transitive unwinder dependency through libc++abi.

**Decision**: Option A — follow deb metadata strictly. `llvm-libunwind1_libs` is a peer of `libc++abi1_libs` under `libc++1_libs`, not a transitive dependency through libc++abi.

### File paths: Single multi-arch path only

**Decision**: Use only `/usr/lib/*-linux-gnu/` paths in LLVM 22 slices. The `/usr/lib/llvm-22/lib/` subdirectory does not exist in 26.04 packages — confirmed by external research (packages.ubuntu.com file lists). This differs from LLVM 20 which had both paths.

### Upgrade chisel before syncing upstream slices

**Decision**: `CHISEL_VERSION` must be upgraded to `v1.4.1` before running `sync-chisel-slices.sh` with `UBUNTU_RELEASE=26.04`. The upstream `chisel.yaml` uses `format: v3` which chisel v1.2.0 cannot parse. Chisel v1.4.1 supports both v2 (custom slices) and v3 (upstream slices).

### Meta-package name libc++1 stays

**Decision**: `libc++1` meta-package name is unchanged. 5 docker-bake.hcl targets reference `libc++1_libs` — keeping the name stable avoids target changes.

### Dockerfile post-install scripts: No changes needed

**Decision**: `install-libcxx.sh` uses glob patterns (`libc++*.so.*.*.*`) that match any LLVM version. All other post-install scripts (tesseract via apt, ladybug/bun via GitHub downloads) are Ubuntu-version-agnostic. No changes required.

## Ordering Constraints

- Slice 1 (infrastructure) MUST come first — chisel version and Ubuntu release drive everything
- Slice 2 (custom slices) MUST follow Slice 1 — needs correct chisel.yaml version/suites for slice validation
- Slice 3 (documentation) MUST follow Slice 2 — needs correct slice names to document
- Chisel upgrade (`CHISEL_VERSION=v1.4.1`) MUST happen before syncing upstream slices

## Verification Notes

- **No automated tests exist** — all verification is manual
- **Static target as canary**: Build `docker buildx bake static` first after Slice 1 to validate chisel infrastructure before touching custom slices
- **libcxx target validates custom slices**: `docker buildx bake libcxx` after Slice 2
- **ldd checks**: `docker run --rm --entrypoint /bin/bash <image> -c "ldd /usr/lib/*-linux-gnu/libc++.so.1"` inside built libcxx image
- **Cross-arch**: Repeat builds for `--set *.platform=linux/arm64`
- **Prior precedent lesson**: LLVM slices took 3 iterations to stabilize (commits `5a03b7a` → `51a9fbd` → `371ede1`). Budget for build-test-fix cycles.
- **Prior precedent lesson**: Even a 1-line version change caused 15-day-undetected README drift (`a1132bd` → `8c4859e`). Verify all 12+ doc references.
- **Prior precedent lesson**: Workflow reordering in `a1132bd` accidentally removed checkout step. Don't touch CI workflows at all.
