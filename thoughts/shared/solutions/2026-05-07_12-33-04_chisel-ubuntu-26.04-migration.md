# Solution Analysis: Migrate chisel images to Ubuntu 26.04 LTS

**Date**: 2026-05-07T12:33:04-0700
**Author**: Ivan Porto Carrero
**Commit**: 266a6ba
**Branch**: main
**Repository**: containers

## Research Question

Migrate chisel-based container images (static, libc, libc-ssl, libcxx, libcxx-ssl, libcxx-ssl-tesseract, libcxx-ssl-ladybug, lbug-cli, bun, fission-bun) from Ubuntu 25.10 to Ubuntu 26.04 LTS (Resolute Raccoon, released 2026-04-23).

## Summary

**Problem**: chisel-based images run on Ubuntu 25.10 (non-LTS, EOL 2026-07). 26.04 LTS is available with 5-year support until 2031.

**Recommended**: Single-version bump to 26.04 ‚Äî update `UBUNTU_RELEASE` default, rewrite 4 custom chisel slices for LLVM 22, upgrade chisel tool to ‚â•v1.4.0 (format v3). Keep netdebug and rustbuilder unchanged initially.

**Effort**: Medium (~3 days)
**Confidence**: Medium

## Problem Statement

**Requirements:**
- All chisel-based images build from `ubuntu:26.04`
- Correct LLVM C++ runtime (libc++, libc++abi, libunwind) for LLVM 22
- Chisel slicing works with 26.04 repository archives
- Image tags reflect `26.04` release

**Constraints:**
- MUST NOT break existing 25.10 consumers ‚Äî tags like `ghcr.io/casualjim/bare:static` will move to 26.04
- rustbuilder stays on 24.04 (separate concern, apt.llvm.org missing resolute)
- netdebug stays on 25.10 for now (full Ubuntu image, can migrate later)
- Must work for both linux/amd64 and linux/arm64

**Success criteria:**
- `docker buildx bake static libc libc-ssl libcxx libcxx-ssl libcxx-ssl-tesseract libcxx-ssl-ladybug lbug-cli bun fission-bun` builds all chisel targets successfully
- Resulting images are functional (not just building ‚Äî runtime libraries resolve)
- Image sizes remain comparable or smaller than 25.10 equivalents

- `Dockerfile:1-2` ‚Äî `ARG UBUNTU_RELEASE=25.10` and `ARG CHISEL_VERSION=v1.2.0`
- `sync-chisel-slices.sh:7` ‚Äî `UBUNTU_RELEASE="${UBUNTU_RELEASE:-25.10}"`
- `chisel-releases/chisel.yaml:8-11` ‚Äî `version: 25.10`, suites `questing*`

**Relevant patterns:**
- One prior Ubuntu version bump: commit `a1132bd` changed to 25.10 ‚Äî single-line change in docker-bake.hcl
- Chisel slice syncs via `sync-chisel-slices.sh` ‚Äî 3 prior sync commits (`5d83207`, `460440f`, `027c0ce`)
- Custom chisel slices: `libc++1.yaml`, `libc++1-20.yaml`, `libc++abi1-20.yaml`, `libunwind-20.yaml` ‚Äî local, maintained outside upstream syncing

**Integration points:**
- `docker-bake.hcl:1` ‚Äî `UBUNTU_RELEASE` variable (28 references across all targets)

## Solution Options

### Option 1: Full 26.04 Migration (Recommended)

**How it works:**
Update `UBUNTU_RELEASE` to `26.04` in docker-bake.hcl, upgrade chisel to v1.4.1, re-sync upstream slices from the `ubuntu-26.04` chisel branch, rewrite 4 custom libc++ slices for LLVM 22 paths, and update all documentation. Package name changes: `libc++1-20` ‚Üí `libc++1` (meta), `libc++abi1-20` ‚Üí `libc++abi1` (runtime), `libunwind-20` ‚Üí `llvm-libunwind1` (runtime).

**Pros:**
- Single variable change for 10+ chisel images ‚Äî low touch surface
- LTS support until 2031 vs 25.10 EOL July 2026
- All upstream packages confirmed available (libssl3t64, libstdc++6, etc.)
- Docker, PostgreSQL apt repos already support "resolute"
- `canonical/chisel-releases` has active `ubuntu-26.04` branch

**Cons:**
- Chisel format v3 breaking change ‚Äî requires CHISEL_VERSION ‚â• v1.4.0
- 3 custom chisel slices have NO upstream equivalents (must be written from scratch)
- No prior precedent for this type of multi-file migration (single-var bump is simpler)
- 25.10 image consumers tag-pinned to `latest` or version-specific tags break silently
- README and docs need ~12+ content updates

**Complexity:** Medium (~3 days)
- Files to modify: 9 (~250 lines)
- Files to create: 3 new custom chisel slice YAMLs (~50 lines)
- Files to delete: 4 old custom chisel slices (libc++1-20, libc++abi1-20, libunwind-20, libc++1 old)
- Risk level: Medium

## Comparison

Single candidate ‚Äî 26.04 migration is the only viable path. 25.10 goes EOL in ~2 months, no LTS option.

## Recommendation

**Selected:** Full 26.04 Migration

**Rationale:**
- 25.10 reaches end-of-life July 2026 ‚Äî waiting means building on an unsupported base
- All critical dependencies verified available (Docker Hub, apt repos, chisel branch)
- One precedent commit (`a1132bd`) shows the UBUNTU_RELEASE bump pattern works
- Chisel format v3 upgrade is required regardless ‚Äî upstream dropped v2 for 26.04

**Trade-offs:**
- Accepting ~2 days of custom slice authoring for libc++/libunwind to avoid waiting for upstream chisel coverage
- Accepting LLVM path changes (20‚Üí22) which ripple into post-install scripts

**Implementation approach:**

1. **Upgrade chisel tool** ‚Äî `docker-bake.hcl:4-5`: change `CHISEL_VERSION` from `v1.2.0` to `v1.4.1`
2. **Update UBUNTU_RELEASE** ‚Äî `docker-bake.hcl:1-3`: change `"25.10"` to `"26.04"`
3. **Re-sync chisel slices** ‚Äî run `sync-chisel-slices.sh` with `UBUNTU_RELEASE=26.04` to pull upstream slices; update the script's default
4. **Rewrite custom libc++ slices**:
   - Delete `chisel-releases/slices/libc++1-20.yaml`, `libc++abi1-20.yaml`, `libunwind-20.yaml`
   - Remove old `libc++1.yaml` meta-slice
   - Create `libc++1.yaml` ‚Äî meta-package depending on `libc++abi1_libs`, `libunwind1_libs`
   - Create `libc++abi1.yaml` ‚Äî content from `/usr/lib/llvm-22/lib/libc++abi.so.1*` and `/usr/lib/*-linux-gnu/libc++abi.so.1*`
   - Create `llvm-libunwind1.yaml` ‚Äî content from `/usr/lib/llvm-22/lib/libunwind.so.1*` and `/usr/lib/*-linux-gnu/libunwind.so.1*`
5. **Update chisel config** ‚Äî `chisel-releases/chisel.yaml:8-11`: suites ‚Üí `resolute, resolute-security, resolute-updates`, version ‚Üí `26.04`
6. **Update post-install scripts** ‚Äî `post-install/install-libcxx.sh:11`: llvm-20 ‚Üí llvm-22 paths
7. **Update documentation** ‚Äî `README.md`: all 25.10 references ‚Üí 26.04
8. **Verify** ‚Äî `docker buildx bake static libc libcxy libcxx-ssl` on both amd64 and arm64

**Integration points:**
- `docker-bake.hcl:1` ‚Äî single variable, 28 references update atomically
- `sync-chisel-slices.sh:7-8` ‚Äî default and upstream branch
- `chisel-releases/chisel.yaml:8-11` ‚Äî archive config
- `chisel-releases/slices/libc++1.yaml` through `llvm-libunwind1.yaml` ‚Äî 3 new, 4 deleted
- `post-install/install-libcxx.sh` ‚Äî path s/llvm-20/llvm-22/

**Patterns to follow:**
- Version bump: `docker-bake.hcl:1-3` ‚Äî same pattern as commit `a1132bd`
- Custom slice format: `chisel-releases/slices/libc++1-20.yaml` ‚Äî follow existing anatomy with updated paths

**Risks:**
- **Upstream libc++ slices missing**: No `libc++1.yaml`, `libc++abi1.yaml`, or `llvm-libunwind1.yaml` in upstream `ubuntu-26.04` branch. Custom slices must be authored. Mitigation: verify against Ubuntu package contents via `dpkg -L` on a real 26.04 system or Docker container.
- **Tag drift for downstream consumers**: `ghcr.io/casualjim/bare:static-25.10` disappears; `latest` moves to 26.04. Mitigation: communicate clearly in release notes.
- **Post-install script path changes**: `install-libcxx.sh` symlink creation depends on correct library paths. Mitigation: test by building `libcxx` target.

## Scope Boundaries
