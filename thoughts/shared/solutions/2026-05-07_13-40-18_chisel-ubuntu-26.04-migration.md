# Solution Analysis: Chisel Ubuntu 26.04 Migration

**Date**: 2026-05-07T13:40:18-0700
**Author**: Ivan Porto Carrero
**Commit**: 266a6ba
**Branch**: main
**Repository**: containers

## Research Question
Update the chisel-based container images (static, libc, libc-ssl, libcxx, libcxx-ssl, libcxx-ssl-tesseract, libcxx-ssl-ladybug, lbug-cli, bun, fission-bun) from Ubuntu 25.10 (Questing) to Ubuntu 26.04 LTS (Resolute Raccoon). Scope: chisel-based images only — rustbuilder and netdebug are out of scope.

## Summary
**Problem**: Ubuntu 25.10 (non-LTS) reaches EOL July 2026. Ubuntu 26.04 LTS was released April 23, 2026 and offers 5+ years of support.
**Recommended**: Chisel-26.04 Migration — single-version bump, no dual builds.
**Effort**: Medium (~2–3 days)
**Confidence**: Medium — core mechanics are well-understood but custom slice rewrite and chisel format v3 upgrade carry execution risk.

## Problem Statement

**Requirements:**
- All 10 chisel-based images must build and run on Ubuntu 26.04
- Library resolution must work: libstdc++6, libssl3t64, libc++1, libc++abi, libunwind, zlib, libgcc-s1
- Tesseract OCR + language packs must install and function
- Ladybug shared library must load correctly
- Bun runtime must pass `bun --version`
- Cross-architecture builds (amd64 + arm64) must succeed

**Constraints:**
- No dual-version builds — drop 25.10 support entirely
- Chisel toolchain must remain the build mechanism
- Custom LLVM libc++ slices must be maintained (not available upstream)
- CI workflows should not need structural changes

**Success criteria:**
- `docker buildx bake` succeeds for all chisel targets on linux/amd64 and linux/arm64
- `ldd` checks pass on all shared libraries inside built images
- Image sizes comparable to 25.10 baselines

## Ubuntu 26.04 Landscape (External Research)

### Release and ecosystem maturity

- **Released**: April 23, 2026 (~2 weeks ago from current date)
- **Docker Hub**: `ubuntu:26.04` available, also `latest`/`rolling` tag
  - Source: https://hub.docker.com/_/ubuntu
- **Codename**: Resolute Raccoon ("resolute")
- **LTS support**: Standard to April 2026, expanded (security) to May 2031, full EOL April 2038
  - Source: https://github.com/canonical/chisel-releases/blob/ubuntu-26.04/chisel.yaml

### Chisel upstream support

- **Branch exists**: `canonical/chisel-releases` has `ubuntu-26.04` branch with 249 commits
  - Source: https://github.com/canonical/chisel-releases/tree/ubuntu-26.04
- **Format**: Uses `format: v3` (current repo uses `format: v2`)
- **Slice coverage**: 100+ slice definitions covering all base packages we use:
  base-files, base-passwd, ca-certificates, tzdata, media-types, bash, coreutils, curl, jq, zlib1g, libssl3t64, libstdc++6, libgcc-s1, openssl
- **No libc++ slices upstream**: `libc++1`, `libc++abi1`, `llvm-libunwind1` are NOT in the upstream branch — must remain custom

### Chisel version compatibility (Critical)

- **Current**: `v1.2.0` — supports format v1 and v2 only
- **v3 format**: Introduced in chisel v1.3.0 (Oct 21, 2025)
- **Latest**: v1.4.1 (Apr 6, 2026)
  - Source: https://github.com/canonical/chisel/releases
- **Action required**: Upgrade `CHISEL_VERSION` from `v1.2.0` to `v1.4.1`

### Urgency

Ubuntu 25.10 (Questing) is a non-LTS release. The current local `chisel.yaml` shows `end-of-life: 2026-07-09` — **approximately 2 months from now**. After that date, the Ubuntu archives for questing may stop receiving updates. While the Docker image and chisel slices would still work, security patches would cease.

## Solution: Single-Version Bump to 26.04

**How it works**: Update `UBUNTU_RELEASE` to `26.04`, upgrade chisel to `v1.4.1`, sync upstream slices, rewrite custom LLVM libc++ slices for LLVM 22, update all documentation references.

### Pros
- **LTS longevity**: 5+ years of security support (vs ~2 months for 25.10)
- **Single-variable change**: `UBUNTU_RELEASE` cascades through 28 tag expressions automatically
- **Upstream chisel slices ready**: 26.04 branch has all base packages — sync script works with a one-line change
- **Chisel already supports 26.04**: Latest chisel release (v1.4.1) fully supports format v3
- **Minimal Dockerfile changes**: Same package names for all non-LLVM libraries

### Cons
- **Custom slice rewrite required**: 3 new slices for LLVM 22 must be hand-authored (no upstream equivalents)
- **Chisel format v3**: Requires upgrading from v1.2.0 to v1.4.1 — format v2 declarations in the local `chisel.yaml` become v3
- **No automated tests**: All verification is manual (~15–20 steps, cross-arch)
- **libc++abi1 unwinder dependency uncertain**: The dependency chain `libc++1 → libc++abi1 → ???` in 26.04 may differ from the LLVM 20 chain (`libc++1 → libc++abi1-20 → libunwind-20`). Package metadata shows `libc++abi1` depends on `libc6` and `libgcc-s1` only — the unwind dependency may be at the deb level but not at the libc++abi1 package level. Needs verification.

### Complexity: Medium (~2 days)

Files to create: 3 (~55 lines of YAML)
- `chisel-releases/slices/libc++abi1.yaml` (~17 lines)
- `chisel-releases/slices/llvm-libunwind1.yaml` (~17 lines)
- New `chisel-releases/slices/libc++1.yaml` (~5 lines — meta-slice rewrite)

Files to delete: 3
- `chisel-releases/slices/libc++1-20.yaml`
- `chisel-releases/slices/libc++abi1-20.yaml`
- `chisel-releases/slices/libunwind-20.yaml`

Files to modify: 7 (~30 lines changed)
- `docker-bake.hcl` — UBUNTU_RELEASE, CHISEL_VERSION
- `Dockerfile` — ARG defaults
- `sync-chisel-slices.sh` — default RELEASE, CUSTOM_PACKAGES array
- `chisel-releases/chisel.yaml` — format, version, suites
- `README.md` — 12+ references
- `docs/adding-custom-slices.md` — version + slice examples

Risk level: Medium

## Comparison

Single candidate — no comparison needed. The alternatives considered and rejected:
- **Dual-build (25.10 + 26.04)**: Rejected by explicit requirement. Would double CI build matrix without benefit since 25.10 EOL is imminent.

## Recommendation

**Selected:** Chisel-26.04 Migration — single-version bump.

**Rationale:**
- Ubuntu 25.10 EOL is July 2026 — migration is time-sensitive
- 26.04 LTS provides 5+ years of security support vs ~2 months remaining
- Upstream chisel support is production-ready (249 commits on ubuntu-26.04 branch)
- All non-LLVM packages keep the same names — the blast radius is confined to custom slices
- Single-variable `UBUNTU_RELEASE` change cascades automatically through tags and args

**Trade-offs:**
- Accepting manual verification burden (no automated tests) for chisel format v3 upgrade safety
- Accepting execution risk on custom slice rewrite for LLVM 22 — mitigated by following the existing pattern from LLVM 20 slices

### Implementation approach

**Phase 1: Chisel infrastructure (critical path)**
1. Run `sync-chisel-slices.sh` with `UBUNTU_RELEASE=26.04` to pull upstream slices into a temp location
2. Copy upstream `chisel.yaml` (format v3) to local `chisel-releases/`
3. Update `docker-bake.hcl`: `UBUNTU_RELEASE="26.04"`, `CHISEL_VERSION="v1.4.1"`
4. Update `Dockerfile`: ARG defaults for UBUNTU_RELEASE and CHISEL_VERSION
5. Verify `static` target builds (exercises chisel tool + base slices only)

**Phase 2: Custom slice rewrite**
1. Spin up `ubuntu:26.04` container, inspect LLVM 22 packages:
   ```bash
   docker run --rm -it ubuntu:26.04 bash
   apt-get update && apt-cache depends libc++1 libc++abi1 llvm-libunwind1
   dpkg --contents /var/cache/apt/archives/libc++abi1_*.deb
   dpkg --contents /var/cache/apt/archives/llvm-libunwind1_*.deb
   dpkg --contents /var/cache/apt/archives/libc++1_*.deb
   ```
2. Write `chisel-releases/slices/libc++abi1.yaml` (pattern: copy `libc++abi1-20.yaml`, update paths/sources)
3. Write `chisel-releases/slices/llvm-libunwind1.yaml` (pattern: copy `libunwind-20.yaml`, update paths/sources/name)
4. Rewrite `chisel-releases/slices/libc++1.yaml` to delegate to new slices
5. Delete old LLVM 20 slices
6. Build `libcxx` target, verify `ldd /usr/lib/*-linux-gnu/libc++.so.1` resolves

**Phase 3: Full matrix verification**
1. Build all chisel targets: `docker buildx bake static libc libc-ssl libcxx libcxx-ssl libcxx-ssl-tesseract libcxx-ssl-ladybug lbug-cli bun fission-bun`
2. Smoke-test each image:
   - `bun`: `docker run --rm ghcr.io/casualjim/bun:latest --version`
   - `lbug-cli`: `docker run --rm ghcr.io/casualjim/lbug-cli:latest lbug --version`
   - `libcxx-ssl-tesseract`: verify tesseract binary + language data
3. Cross-arch: repeat for `--set *.platform=linux/arm64`
4. Compare image sizes against 25.10 baselines

**Phase 4: Documentation**
1. Update `README.md` — 12+ references from "25.10" to "26.04"
2. Update `docs/adding-custom-slices.md` — version and slice examples
3. Update `.agents/skills/add-chisel-slice/SKILL.md` — version and path examples
4. Update `sync-chisel-slices.sh` — default RELEASE and CUSTOM_PACKAGES

### Integration points
- `docker-bake.hcl:2` — `UBUNTU_RELEASE` default
- `docker-bake.hcl:5` — `CHISEL_VERSION` default
- `Dockerfile:1-2` — ARG defaults
- `chisel-releases/chisel.yaml:1,9,11` — format, version, suites
- `sync-chisel-slices.sh:7,53-57` — default RELEASE, CUSTOM_PACKAGES

### Patterns to follow
- LLVM 20 custom slice format (`chisel-releases/slices/libc++abi1-20.yaml`) as template for LLVM 22 slices
- `sync-chisel-slices.sh:19-51` — REQUIRED_PACKAGES array structure for upstream dependency resolution

### Risks
- **Chisel v1.2.0 + format v3 incompatibility**: If v1.4.1 has regression, builds fail. **Mitigation**: test `static` target first, which has no custom slices, before touching LLVM slices.
- **libc++abi1 unwind dependency**: The LLVM 22 package dependency chain may differ. **Mitigation**: verify with `apt-cache depends` during Phase 2, adjust slice essentials accordingly.
- **No rollback safety**: No automated tests to catch regressions. **Mitigation**: commit incrementally (infrastructure → slices → docs), verify each step before proceeding.

## Follow-up Analysis 2026-05-07T13:50:00-0700

**LLVM 22 alignment confirmed by Rust toolchain.**

Rust 1.95.0 (2026-04-14) uses LLVM 22.1.2 — the exact same version that ships in Ubuntu 26.04 (`llvm-toolchain-22`, `1:22.1.2-1ubuntu1`). This is meaningful because:

1. **No LLVM version skew**: When rustbuilder is eventually migrated to 26.04, the system LLVM (from Ubuntu packages) and the LLVM bundled with rustc will be the same major version. This eliminates the current mismatch where rustbuilder pins clang-21 from apt.llvm.org on a 24.04 base.

2. **libc++/libc++abi ABI compatibility**: Applications compiled with Rust 1.95.0 (which may link against system libc++) will be ABI-compatible with the `libc++abi1` and `llvm-libunwind1` packages from Ubuntu 26.04 — all from LLVM 22.1.2.

3. **rustbuilder migration path simplified**: When rustbuilder moves to 26.04 (separate effort, currently blocked by apt.llvm.org availability for resolute), we can drop the `apt.llvm.org` dependency and use Ubuntu-native LLVM 22 packages directly, matching what chisel-based images already package.
